# Milestone Boundaries

## Purpose
Keep work grouped into clear, reviewable milestones.

## Use When
Starting, resuming, or preparing to commit a multi-file change.

## Practice
- Define the milestone in one sentence.
- Name in-scope file groups before editing.
- Keep unrelated docs, code, CI, and git workflow changes separate.
- Let validation scale with the milestone's risk.

## Anti-Patterns
- Combining feature work, cleanup, docs, and CI in one commit.
- Expanding scope because nearby files are visible.
- Treating a handoff or summary as approval for unrelated work.

## Validation Or Report Expectations
- Report milestone scope, changed files, validation, risks, and next safe action.
